          Hi @EdBWA, here are some screenshots for the GUI part of the config:

           

          First, in the Controller Settings, click on Firewall & Routing in the left column. Then at the top, click on Firewall -> Groups. Then click the + CREATE NEW GROUP button. Configure your new group like this (you can name it whatever you want):

           

           

          This group is the list of network addresses that we expect to receive IGMP broadcasts from. I just identified a block of addresses that I had read elsewhere were used by CenturyLink. It could almost certainly be done more efficiently, but this works for me. If it doesn't work for you, you can look up on dslreports or other places to find what addresses CL is using in your area. Alternatively, you could just totally open your firewall to all incoming UDP traffic until you determine the proper source address, but I certainly wouldn't leave it that way.

           

          Once you are finished, your Groups page will look like this:

           

           

          Now we need to add the two firewall rules. The first rule will allow IGMP packets to hit the WAN port. Navigate at the top to Firewall -> Rules -> WAN LOCAL and click the + CREATE NEW RULE button. Configure the rule like this (again, name it what you like):

           

           

           

          Once you are finished, your WAN_LOCAL rules will look like this on a default setup:

           

           

          The second rule will allow UDP packets originating from the source group that we configured in the beginning to pass through to the LAN. Navigate to Firewall -> Rules -> WAN IN and click the + CREATE NEW RULE button again. Configure the rule like this (following whatever naming conventions you've been using):

           

           

          When you are finished, your WAN_IN rules will look like this on a default install:

           

           

          That's all there is to it for the GUI part of the config (step 1 in my earlier post).

           

          Now you need to enable the IGMP proxy on the gateway (step 2 in my earlier post). This cannot be done through the GUI at this time.

           

          To do this, you need to place a custom config.gateway.json file in the appropriate site directory on your controller. Using your text editor of choice, create a file named config.gateway.json with the following contents:

           

          {
        "protocols": {
                "igmp-proxy": {
                        "interface": {
                                "eth0": {
                                        "alt-subnet": [ "0.0.0.0/0" ],
                                        "role": "upstream",
                                        "threshold": "1"
                                        },
                                "eth1": {
                                        "alt-subnet": [ "0.0.0.0/0" ],
                                        "role": "downstream",
                                        "threshold": "1"
                                        }
                                     }
                              }
                     }
}

          Note that you will probably need to modify this since you suspect that CenturyLink requires VLAN 201 tags in your area. In this case, after enabling WLAN tagging to VLAN 201 in the GUI settings for your USG, you will want to change "eth0" (the upstream) to "eth0.201" in the json file. Also note that if you have a USG-Pro, the default interfaces are different (I think WAN is eth2 and LAN is eth0 by default on the Pro).

           

          You will need to copy this file into the appropriate directory on the controller. You can refer to the instructions here for where to place the file.

           

          Now all you need to do is reprovision your gateway. It will automatically start igmp-proxy on every reprovision.

           

          Good luck!

          @panthergm Hmmm. Let's try to reduce the variables. I am using USG firmware 4.3.33.4936086 and controller 5.4.8 (testing). Do you have the same firmware? And as for the lack of VLAN tagging, I guess we're just lucky here in Vegas.

           

          You've probably tried all this, but here's the order I would go in:

           

          1a. For the firewall, I assume you already have accept all IGMP on WAN_LOCAL.

           

          1b. Just to remove the possibility that your source addresses for Prism's multicast are different / wrong, let's temporarily open up all UDP traffic to WAN_IN. From the GUI, make this your first WAN_IN rule:

           

           

          Note that the fields for Source Address Group and Destination Address Group need to be filled in as "Select Group" so that it won't try to match anything. (Or you could just manually insert the rule at the CLI). If it works, you probably want to go here and enable logging so you can find the correct source multicast address by watching dmesg as the packets roll in.

           

          Now SSH into the gateway and issue

           

          show firewall name WAN_LOCAL

          Your top rule should be 2000 accept igmp (with no other conditions). Then issue

           

          show firewall name WAN_IN

          Your top rule should be 2000 accept udp (with no other conditions).

           

          2. Next, let's see what happens when you try to manually invoke the igmp-proxy on the gateway. I am assuming that your WAN (upstream) interface is eth0.201 and your LAN is eth1. If you are using a USG-Pro, these should be changed appropriately.

           

          To eliminate another variable, remove the config.gateway.json and reprovision. Then SSH into the gateway and issue the following:

           

           

          configure
edit protocols igmp-proxy
set interface eth0.201 role upstream
set interface eth1 role downstream
set interface eth0.201 alt-subnet 0.0.0.0/0
set interface eth1 alt-subnet 0.0.0.0/0
set interface eth0.201 threshold 1
set interface eth1 threshold 1
commit
top
save
exit

           

          3. Now assuming you didn't get any errors so far, let's see if anything is coming in. Do

           

          show ip multicast interfaces

          and

           

          show ip multicast mfc

           

          Hopefully, it will either work or reveal some more information on where it is screwing up.

           

          Edit: Fixed ridiculous numbering, sort of.

          Congrats, @Lucebare!

           

          However, it seems like you followed the debugging instructions. You will want to do two things now:

           

          First, you need to close your gateway to unneccessary UDP traffic. Right now, you are wide open to the whole world instead of just to your IPTV provider's broadcasts. To make the WAN_IN rule a bit more tight, we need to figure out that source address. So ssh into your gateway and type

           

          show ip multicast mfc

          Now, in the output, let's look to find the right ip address(es) for your IPTV provider. You want to look for IP addresses in the list that are (1) public address(es), but (2) not your WAN public address, and (3) are in the Origin column.

           

          In my case, CenturyLink appears to be broadcasting from two addresses in the 69.179.0.0/16 subnet. So I went back to Firewall -> Groups and redefined the allowed group to include just the 69.179.0.0/16 subnet. Then I checked my WAN_IN rule to make sure that this group was listed as the permitted source network. Everything still works and now the firewall is a little more sensical.

           

          I'm sure my firewall rules could be a lot more elegantly refined (and someone will probably come along with better ones) but I wouldn't leave your gateway wide open to UDP traffic from all and sundry in the meantime.

           

          Secondly, you need to use a config.gateway.json (as described in earlier posts) to turn on the igmp-proxy in the gateway. Otherwise, your configuration will not survive a reboot or reprovision and you will have to manually ssh into the gateway and restart igmp-proxy every time you reboot or reprovision.

           

          edit: And yeah, like you mentioned, enabling IGMP snooping for the network is a good idea so that your access points don't get hammered with needless broadcasts.

          @calvarezUSA, it seems like you have two issues: (1) how to determine what to put in your config.gateway.json, and (2) how to put the config.gateway.json on your CloudKey so that it is passed to your gateway when it is reprovisioned.

           

          (1) How to determine what to put in your config.gateway.json

           

          First, follow the previously-posted instructions on configuring your firewall (accepting IGMP on WAN_LOCAL and accepting UDP on WAN_IN, both of which can be done in the GUI). For now, you should leave UDP wide open without restricting it to a source network (you should go back and tighten things up once you know that it is working).

           

          You can ssh into your router by using an ssh client like PuTTY for Windows or by typing ssh your.usg.local.ip.address on a Linux / UNIX command line and entering your administrator username / password. Once you're in, you want to manually start igmp-proxy to make sure that it works before creating the config.gateway.json. Assuming you are using a default config with a regular USG and your provider does not require VLAN tagging on the WAN, these commands would be:

           

          configure
edit protocols igmp-proxy
set interface eth0 role upstream
set interface eth1 role downstream
set interface eth0 alt-subnet 0.0.0.0/0
set interface eth1 alt-subnet 0.0.0.0/0
set interface eth0 threshold 1
set interface eth1 threshold 1
commit
top
save
exit

           

          After doing this, your IPTV should (hopefully) be working. If it is, the next step is to create a config.gateway.json file to make sure that igmp-proxy is configured automatically on your USG every time that it is reprovisioned. If it is not working, we need to figure out the problem before bothering to make the configuration permanent -- there is no point in making a config.gateway.json yet and it will only make things more complicated.

           

          (2) How to put the config.gateway.json on your CloudKey

           

          (2a) Creating the config.gateway.json file. If the commands for configuring igmp-proxy above worked, we now want to create a config.gateway.json to do it automatically on reprovisioning. The contents of the file that will automatically configure igmp-proxy in the same way as in step (1) are:

           

          {
        "protocols": {
                "igmp-proxy": {
                        "interface": {
                                "eth0": {
                                        "alt-subnet": [ "0.0.0.0/0" ],
                                        "role": "upstream",
                                        "threshold": "1"
                                        },
                                "eth1": {
                                        "alt-subnet": [ "0.0.0.0/0" ],
                                        "role": "downstream",
                                        "threshold": "1"
                                        }
                                     }
                              }
                     }
}

          If you need to make any changes, you should run your code through a JSON validator like the one here to guard against any accidental formatting changes. Once you are satisfied, paste the code into a text editor and save it to your desktop as config.gateway.json.

           

          (2b) Determining where to put your config.gateway.json file. (Note that I don't have a CloudKey myself, but this should work.) First, you need to identify the right location for the config.gateway.json file. To do this, open your web browser, browse to your controller, and go to the dashboard for your site. Look in the URL on your browser, which should be something like:

           

          https://your.controller.address:8443/manage/site/your_site_name/dashboard 

           

          Take note of the value of your_site_name in the URL. If you have only one site, it will likely be default. If you have multiple sites, it may be a string of alphanumeric characters like a1bcdefg.

           

          (2c) Putting the config.gateway.json file on the controller/CloudKey. Now connect to your CloudKey with an scp client (like WinSCP for Windows) by specifying your CloudKey's local IP address and entering your username and password. Browse to /srv/unifi/data/sites (note that if you are not using a CloudKey, the location will be different, like /var/lib/unifi/data/sites if the controller is on an Ubuntu server).

           

          Look for a subfolder named the same as your_site_name that you discovered from step (2b) above (again, if you have only one site, it should be default). If the folder doesn't exist, create it (make sure it is exact -- case matters) so that you now have a folder named /usr/srv/unifi/data/sites/your_site_name.

           

          Now transfer the config.gateway.json that you saved on your desktop in step (2a) into the /usr/srv/unifi/data/sites/your_site_name folder.

           

          Note: If you are using Linux / UNIX, you can transfer the file from your command line with this command:

           

          scp /your/local/path/to/config.gateway.json your.local.cloudkey.ip.address:/usr/srv/unifi/data/sites/your_site_name

           

          That's it. Now test it by reprovisioning your gateway. The igmp-proxy should be started automatically. If everything works, you should go back and tighten up your firewall rules so that you are only letting in UDP from the necessary address(es) and enable igmp-snooping on your network to cut down on unnecessary broadcast traffic through your LAN.

          @EdBWA - Well, it looks like it's at least partially working. When the stream is first sent out to the set-top boxes, it is sent as UDP unicast for the first 20 or 30 seconds. That gives the STB time to subscribe to the multicast UDP stream.

           

          The initial UDP stream is clearly getting through because you are getting that first 20 seconds worth of TV. So either your STB isn't able to subscribe to the multicast stream, or multicast isn't coming through at all.

           

          On the strength of this reasoning, I suspect the problem is with the igmp-proxy but maybe there is something else on the network that is blocking multicast.

           

          I'd start by removing as many variables as possible. Disable igmp-snooping for now (that is a nice-to-have feature that prevents the multicast from saturating your network, particularly your APs, but is an extra complication while debugging) and put your STB on the same switch as the gateway. Also, if feasible, I'd put your CenturyLink ONT onto WAN1 instead of WAN2 (maybe there's a weird default configuration for WAN2 on the USG-Pro that we don't know about).

           

          Based on your configuration, your igmp-proxy upstream should be eth3.201 (or eth2.201 if you connect the ONT to WAN1, I guess) and your downstream should be eth0. Configure igmp-proxy with these -- are you getting any errors when you invoke it?

           

          Then issue show firewall name WAN_IN and show firewall name WAN_LOCAL. Are your two firewall rules matching any traffic? Based on what you've described, the WAN_IN should definitely be matching some packets. If the WAN_LOCAL rule is not matching anything, maybe something else is blocking IGMP packets from your STB to the multicast group (maybe an evil switch that is doing igmp snooping too enthusiastically).

           

          Also issue show ip multicast mfc on the CLI and let's see what you're getting. (You can post it here if you like, but obscure your public IP.) If things are working right, you should see some public ip address(es) (not your own, but CenturyLink's) that are sending packets to the multicast group.

           

          I'm far from being any sort of expert on this stuff. I hope any of the real McCoys who might be reading this feel free to step in.

          @EdBWA -- No, I am also using a Unifi switch -- it's not that.

           

          While testing, maybe you should consider restarting instead of just reprovisioning your USG-Pro between tests. I don't know if there are any configuration changes (such as ones to igmp-proxy) that might be persistent through a reprovision.

           

          We are edging closer to that point where twisting the dials is getting progressively less likely to help. Soon we will have to start logging packets to see just what is going on. Unfortunately, that will entail real work, so we could give another stab at being lazy.

           

          Right now, I'd like to know if the problem is the firewall. So one thing you could do is just turn the firewall off entirely. Set the top rule for WAN_IN, WAN_OUT and WAN_LOCAL to accept all (all protocols). This is, of course, bending your network over the metaphorical pool table, but if it works, it will be easier to tighten things down afterwards than to try to diagnose where to loosen them in advance.

           

          If it still doesn't work after opening up the firewall, it starts to point the finger at igmp-proxy, but it doesn't prove it. I hope CenturyLink isn't doing something sinister like tagging the multicast packets to a different VLAN or something.
